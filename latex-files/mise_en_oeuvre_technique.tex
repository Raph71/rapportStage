\chapter{Mise en œuvre de la solution}

\section{Architecture cible}

La solution que j'ai conçue s'appuie sur une chaîne d'alerting entièrement pilotée par le code. Les
métriques d'infrastructure sont surveillées via CloudWatch, tandis que les métriques d'application sont
collectées par les collecteurs OpenTelemetry et stockées dans AMP. 
Les règles d'alerte, décrites en Terraform, sont déployées automatiquement sous forme d'alarmes
dans CloudWatch et de règles Prometheus dans AMP. Lorsqu'une alerte se déclenche, elle est transmise
à incident.io, les deux sources suivant un chemin légèrement différent. Depuis AMP, l'Alertmanager
managé transmet directement l'alerte, déjà porteuse de tout son contexte (priorité, description, liens)
grâce aux \emph{labels} de la règle Prometheus. Depuis CloudWatch, l'alarme — bien plus sommaire —
est d'abord publiée sur un topic SNS, puis une fonction Lambda l'enrichit du même contexte avant de la
transmettre à incident.io, afin que les deux sources produisent des notifications homogènes.

C'est ensuite incident.io qui prend en charge le regroupement (grouping), le routage et l'escalade des
alertes, avant de notifier les bonnes personnes sur le canal le plus adapté (Slack, SMS, appel,
notification push ou e-mail).

\section{Le module Terraform et les fichiers de configuration}

Pour industrialiser la création des alertes, j'ai développé un module Terraform unique, réutilisable
par toutes les équipes. Ce module sait créer des alertes aussi bien dans CloudWatch que dans
Prometheus, sans que les développeurs aient besoin de connaître le fonctionnement interne de ces
outils.

Il propose deux types d'alertes prêtes à l'emploi, correspondant aux méthodes présentées au chapitre
précédent : des alertes \textbf{USE} (santé des ressources machine : CPU, mémoire, erreurs 4XX/5XX,
temps de réponse...) et des alertes \textbf{RED} (expérience utilisateur : taux d'erreur, latence...). Si un
besoin plus spécifique se présente, une équipe peut également déclarer une \textbf{alerte
personnalisée}, en écrivant elle-même sa requête Prometheus (PromQL) ou sa configuration
CloudWatch.

Chaque équipe possède son propre dossier dans le dépôt du module et y déclare ses alertes, dans le
fichier correspondant à l'environnement concerné (production, développement ou pré-production).
Cette organisation centralise la déclaration des alertes à un seul endroit, alors qu'elle était auparavant
dispersée entre plusieurs mécanismes.

\subsection{Exemple d'alerte USE par défaut}

L'exemple ci-dessous active, pour le service \texttt{account-options} du cluster ECS
\texttt{clu-betting-api}, les alertes par défaut de type USE (CPU, mémoire, erreurs 4XX/5XX, temps de
réponse) : il suffit de déclarer ces champs vides pour que le module applique automatiquement des
seuils par défaut.

\begin{lstlisting}[style=codestyleblack, language=HCL, caption={Activation d'alertes USE par défaut sur un service ECS}, label={lst:use-example}]
ecs = {
  "clu-betting-api" = {
    "account-options" = {
      cpu                  = {}
      memory               = {}
      target_4xx           = {}
      target_5xx           = {}
      target_response_time = {}
    }
  }
}
\end{lstlisting}

\subsection{Exemple d'alerte RED par défaut}

De la même manière, l'exemple suivant active les alertes par défaut de type RED (taux d'erreur et
latence) pour une API du même service.

\begin{lstlisting}[style=codestyleblack, language=HCL, caption={Activation d'alertes RED par défaut sur une API}, label={lst:red-example}]
apis = {
  "clu-betting-api" = {
    "account-options" = {
      error_rate  = {}
      latency_p99 = {}
    }
  }
}
\end{lstlisting}

\subsection{Exemple d'alerte personnalisée}

Lorsque les alertes par défaut ne suffisent pas, une équipe peut déclarer une alerte entièrement
personnalisée, en fournissant sa propre requête PromQL ainsi que le niveau de priorité, la description
et les liens (dashboard, runbook) qui apparaîtront dans la notification finale.

\begin{lstlisting}[style=codestyleblack, language=HCL, caption={Déclaration d'une alerte personnalisée en PromQL}, label={lst:custom-example}]
custom_alerts = {
  "vpn_sogei_contract_limit" = {
    prometheus = {
      promql       = "node_nf_conntrack_entries{instance=~'vpn-sogei.*'}/node_nf_conntrack_entries_limit{instance=~'vpn-sogei.*'}*100>80"
      for_duration = "5m"
    }
    priority      = "Critical"
    description   = "Conntrack entries are above 80% of the limit"
    dashboard_url = "DASHBOARD_URL"
    runbook_url   = "RUNBOOK_URL"
  }
}
\end{lstlisting}

Cet exemple illustre concrètement la notification étudiée plus loin (figure~\ref{fig:incident-sms}) : la
priorité, la description ainsi que les liens vers le dashboard et le runbook déclarés ici sont précisément
ceux qui apparaissent dans la notification reçue par la personne d'astreinte.

Une fois leurs modifications réalisées, les développeurs soumettent une Pull Request, ce qui déclenche
le processus de CI/CD appliquant les changements. 

\section{La chaîne de routage des notifications}

Une fois qu'une alerte se déclenche dans CloudWatch ou dans AMP, elle est transmise à incident.io, qui
applique une logique de notification complète. Nous avons retenu cet outil car il accepte plusieurs sources d'alertes et propose les fonctionnalités suivantes, essentielles pour notre cas d'usage :

\begin{itemize}
	\item \textbf{regrouper} les alertes liées à un même service afin d'éviter la submersion (grouping) ;
	\item \textbf{router} chaque alerte vers les bonnes personnes selon le service impacté, sa criticité et l'environnement ;
	\item \textbf{escalader} automatiquement vers le niveau d'astreinte supérieur toute alerte critique non prise en charge dans les temps ;
	\item \textbf{mettre en sourdine} (silencing) les notifications pendant les maintenances planifiées ;
	\item \textbf{enrichir} chaque notification d'un contexte immédiatement exploitable : niveau de priorité, description de l'alerte, lien vers le runbook et vers le dashboard Grafana du service.
\end{itemize}

Cette étape est décisive : c'est elle qui transforme un signal technique brut en une notification claire,
priorisée et directement actionnable par la personne d'astreinte.

Concrètement, lorsqu'une alerte se déclenche aujourd'hui, la personne d'astreinte reçoit une
notification push envoyée par incident.io directement sur son téléphone, comme illustré à la

Cette notification tranche nettement avec le simple SMS décrit au chapitre précédent
(figure~\ref{fig:sms-astreinte}), qui ne contenait que le nom brut de l'alarme. Elle regroupe désormais
en un seul endroit tous les éléments nécessaires à une première évaluation de l'incident :

\begin{itemize}
	\item une \textbf{description} explicite du problème (ici, la coupure d'un VPN et le composant précis concerné), qui évite d'avoir à interpréter un nom d'alarme technique ;
	\item un \textbf{niveau de priorité} (\emph{Critical} dans cet exemple), qui indique immédiatement le degré d'urgence de l'intervention et la légitimité du réveil ;
	\item un lien direct vers le \textbf{dashboard} Grafana du service concerné, pour visualiser sans délai l'état du système ;
	\item lorsqu'il est configuré pour l'alerte, un lien vers le \textbf{runbook} associé, qui propose une première piste de diagnostic et de résolution ;
	\item le chemin d'escalade suivi (\emph{Path}) ainsi que les actions directement disponibles (accuser réception, déclarer un incident, se signaler indisponible, ou reporter la notification).
\end{itemize}

Ce niveau de contexte permet à la personne d'astreinte de juger en quelques secondes de la criticité
réelle de la situation, sans avoir à se connecter à plusieurs outils pour reconstituer l'information,
contrairement à ce qu'imposait l'ancien système.

\section{Stratégie de transition}

La mise en œuvre d'un nouveau modèle d'alerting en production impose une approche incrémentale,
afin de limiter les risques de régression et de faciliter l'appropriation par les équipes. Trois phases ont
été définies :

\begin{itemize}
	\item \textbf{Phase 1 -- Déploiement de la fondation technique} : mise en place de l'infrastructure et du code nécessaires à l'implémentation complète de la solution.
	\item \textbf{Phase 2 -- Coexistence et pilotage (double-run)} : l'ancien système n'est pas immédiatement désactivé ; la nouvelle solution s'exécute en parallèle sur le service d'une équipe pilote, ce qui permet de comparer les alertes émises par les deux systèmes et d'accompagner les développeurs dans l'ajustement des seuils.
	\item \textbf{Phase 3 -- Déclassement de l'ancien système} : une fois la fiabilité de la nouvelle solution démontrée, les alertes de l'ancienne solution sont mises en sourdine, sa visualisation est maintenue temporairement pour observation, avant sa désactivation définitive par la suppression des anciennes Lambdas.
\end{itemize}

Cette trajectoire prudente instaure une confiance progressive dans la nouvelle solution et évite tout
angle mort pendant la bascule.

\section{Ownership et responsabilisation des équipes}

Au-delà de l'aspect technique, la solution corrige l'une des limites majeures du système existant :
l'absence d'ownership des alertes par service. Chaque alerte porte désormais un tag \texttt{Team} et
un tag \texttt{Service} qui l'associent sans ambiguïté à l'équipe responsable, et le routage s'appuie sur
ces tags pour diriger la notification vers le canal Slack de cette équipe plutôt que vers un canal
générique.

Cette responsabilisation passe concrètement par le mécanisme de Pull Request : les équipes ajustent
elles-mêmes, dans leur fichier \texttt{.tfvars}, les seuils et l'activation des alertes de leur service, sous
revue de l'équipe Infrastructure. Chaque alerte critique est en outre associée à un runbook et à un
dashboard Grafana propres au service, ce qui place l'équipe propriétaire en première ligne du
diagnostic — condition nécessaire à une gouvernance durable de l'alerting.

en œuvre permet de mesurer concrètement les bénéfices apportés sur chacun des points faibles
identifiés lors du diagnostic.
